이전

Yacht Auction

NYPC CODE BATTLE · 온라인 라운드

문제 설명

버섯 게임을 끝낸 예티와 핑크빈은 이제 그란디스로 넘어와, 럭키 다이스 스킬을 쓰며 주사위를 던지던 과거를 회상하며 Yacht Auction이라는 보드게임을 하기로 했습니다.

경기 규칙

  • 한 경기는 131313번의 라운드로 이루어집니다.
  • 각 라운드는 입찰 단계와 점수 획득 단계로 이루어집니다.
    • 예외적으로, 첫번째 라운드에서는 점수 획득 단계를 진행하지 않고, 마지막인 131313번째 라운드에서는 입찰 단계를 진행하지 않습니다.

입찰 단계

입찰 단계는 다음과 같이 진행됩니다.

  1. AAA묶음 주사위 555개와 BBB묶음 주사위 555개를 굴립니다. 각 주사위는 111 이상 666 이하의 정수가 동일한 확률로 서로 독립적으로 나오는 주사위입니다.
  2. 각 사람은 자신이 가져가고 싶은 주사위 묶음 AAA 혹은 BBB와 입찰에 사용할 000 이상 100 000100\,000100000 이하의 입찰 점수를 제출합니다.
  3. 제출한 주사위 묶음과 입찰 점수를 통해 다음 규칙에 따라 주사위를 가져갑니다.
  • 서로 가져가고 싶은 주사위 묶음이 다른 경우 각자가 원하는 주사위 묶음을 가져갑니다.
  • 서로 가져가고 싶은 주사위 묶음이 같은 경우 더 높은 점수를 제출한 사람이 해당 묶음을 가져갑니다.
  • 만약 제출한 묶음과 입찰 점수가 모두 같다면 주사위를 가져가는 사람을 무작위로 정합니다. 각 사람이 선택될 확률은 같으며, 매 시행은 독립입니다.
  1. 이후 다음 규칙에 따라 점수를 계산합니다.
  • 자신이 가져가고 싶은 주사위 묶음을 가져간 경우, 입찰 점수만큼 점수를 잃습니다.
  • 자신이 가져가고 싶은 주사위 묶음을 가져가지 못한 경우, 입찰 점수만큼 점수를 얻습니다.

점수 획득 단계

가지고 있는 주사위 101010개(단, 마지막 점수 획득 단계에서는 555개)의 주사위 중 원하는 주사위를 555개 선택합니다. 이후, 다음 점수 규칙중 하나를 정해서 점수를 얻습니다. 한 경기에서 한 종류의 점수 규칙은 최대 한 번만 사용할 수 있습니다.

기본 점수 규칙

  • ONE: 1이 나온 주사위에 적힌 수의 합×1 0001\text{이 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}1이 나온 주사위에 적힌 수의 합×1000점
  • TWO: 2가 나온 주사위에 적힌 수의 합×1 0002\text{가 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}2가 나온 주사위에 적힌 수의 합×1000점
  • THREE: 3이 나온 주사위에 적힌 수의 합×1 0003\text{이 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}3이 나온 주사위에 적힌 수의 합×1000점
  • FOUR: 4가 나온 주사위에 적힌 수의 합×1 0004\text{가 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}4가 나온 주사위에 적힌 수의 합×1000점
  • FIVE: 5가 나온 주사위에 적힌 수의 합×1 0005\text{가 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}5가 나온 주사위에 적힌 수의 합×1000점
  • SIX: 6이 나온 주사위에 적힌 수의 합×1 0006\text{이 나온 주사위에 적힌 수의 합} \times \mathbf{1\,000}6이 나온 주사위에 적힌 수의 합×1000점

또한, 기본 점수 규칙에서 63 00063\,00063000점 이상을 얻은 경우 보너스 점수 35 000\mathbf{35\,000}35000점을 추가로 얻게 됩니다.

조합 점수 규칙

  • CHOICE: 다섯 개의 주사위에 적힌 모든 수의 합×1 000\textbf{다섯 개의 주사위에 적힌 모든 수의 합} \times \mathbf{1\,000}다섯 개의 주사위에 적힌 모든 수의 합×1000점
  • FOUR_OF_A_KIND: 같은 수가 적힌 주사위가 444개 있다면, 다섯 개의 주사위에 적힌 모든 수의 합×1 000\textbf{다섯 개의 주사위에 적힌 모든 수의 합} \times \mathbf{1\,000}다섯 개의 주사위에 적힌 모든 수의 합×1000점. 없을 경우 0\mathbf{0}0점
  • FULL_HOUSE: 333개의 주사위에 적힌 수가 서로 같고, 다른 222개의 주사위에 적힌 수도 서로 같으면 다섯 개의 주사위에 적힌 모든 수의 합×1 000\textbf{다섯 개의 주사위에 적힌 모든 수의 합} \times \mathbf{1\,000}다섯 개의 주사위에 적힌 모든 수의 합×1000점, 아닐 경우 0\mathbf{0}0점
  • SMALL_STRAIGHT: 444개의 주사위에 적힌 수가 1234,2345,34561234, 2345, 34561234,2345,3456중 하나로 연속되어 있을 때, 15 000\mathbf{15\,000}15000점, 아닐 경우 0\mathbf{0}0점
  • LARGE_STRAIGHT: 555개의 주사위에 적힌 수가 12345,2345612345, 2345612345,23456중 하나로 연속되어 있을 때, 30 000\mathbf{30\,000}30000점, 아닐 경우 0\mathbf{0}0점
  • YACHT: 555개의 주사위에 적힌 수가 모두 같을 때 50 000\mathbf{50\,000}50000점, 아닐 경우 0\mathbf{0}0점

최종 점수 및 승리 조건

모든 라운드가 끝난 이후, 총점이 높은 사람이 승리합니다. 총점이 같다면 무승부입니다.

  • 플레이어의 총점=기본 점수 규칙에 따른 점수+조합 점수 규칙에 따른 점수+입찰 점수\text{플레이어의 총점} = \text{기본 점수 규칙에 따른 점수} + \text{조합 점수 규칙에 따른 점수} + \text{입찰 점수}플레이어의 총점=기본 점수 규칙에 따른 점수+조합 점수 규칙에 따른 점수+입찰 점수

입/출력 형식

참가자의 프로그램은 한 줄씩 입력을 받아 각 명령에 따라 동작해야 합니다. 출력이 필요한 명령은 아래 표에서 빨간색으로 시간 제한과 함께 표시되어 있습니다. 이 시간 제한 안에 출력을 하지 못한 경우 시간 초과 (TLE) 판정을 받게 되며, 해당 게임에서 실격패 처리됩니다.

테스팅 툴 (GUI)

참가자가 작성한 프로그램을 테스팅하기 위한 테스팅 툴인 testing-tool-yacht가 제공됩니다. 참가자의 환경에서 컴파일한 실행 파일 혹은 참가자의 실행 환경의 python, java, node, lua 환경과 스크립트를 이용하여 로컬에서 문제를 테스트해 볼 수 있습니다.

  • 64비트 Windows (x86-64) (다운로드)
    • 다운로드받은 .exe 파일을 실행합니다.
  • MacOS Universal (다운로드)
    • .dmg 파일을 더블 클릭한 이후 testing-tool-yacht 파일을 Applications 폴더에 드래그해서 복사합니다. 그 이후 testing-tool-yacht 앱을 실행합니다.
    • 사용자의 보안 설정은 App Store에서 다운로드한 응용 프로그램의 설치만 허용합니다. 라는 안내가 나오는 경우, "설정 > 개인정보 및 보안"에서 앱의 정보를 확인합니다.
  • deb (Ubuntu/Debian AMD64) (다운로드)
    • apt 혹은 dpkg을 이용해 .deb 파일을 설치한 이후 testing-tool-yacht을 실행합니다.
  • rpm (Fedora/RHEL x86-64) (다운로드)
    • rpm 혹은 yum을 이용해 .rpm 파일을 설치한 이후 testing-tool-yacht을 실행합니다.

테스팅 툴 (CLI)

참가자가 작성한 프로그램을 테스팅하기 위한 테스팅 툴 패키지인 testing-tool-yacht-cli.zip (다운로드) 가 주어집니다. 자세한 내용은 파일 안에 있는 README.md를 참고하세요.

CLI 테스팅 툴을 수정하는 것은 자유이며, 실행 방법과 관련해서는 다른 사람과 자유롭게 토론할 수 있습니다. 하지만 수정된 테스팅 툴 혹은 테스팅 툴이 출력한 로그 파일 등을 팀원 이외의 사람과 공유하거나, 이와 관해 토론해서는 안됩니다.

채점 결과에서는 테스팅 툴이 출력한 로그와 같은 형식의 로그 파일을 받아볼 수 있습니다. 단, 실제 채점 프로그램은 테스팅 툴과 다르게 구현되어 있으므로, 테스팅 툴에서의 정상 작동이 실제 채점에서의 정상 작동을 항상 의미하지는 않습니다.
